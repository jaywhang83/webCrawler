<p><b>Key exchange</b> (also <b>key establishment</b>) is a method in <a href="/wiki/Cryptography" title="Cryptography">cryptography</a> by which <a href="/wiki/Cryptographic_keys" class="mw-redirect" title="Cryptographic keys">cryptographic keys</a> are exchanged between two parties, allowing use of a cryptographic <a href="/wiki/Algorithm" title="Algorithm">algorithm</a>.
</p><p>If the sender and receiver wish to exchange encrypted messages, each must be equipped to encrypt messages to be sent and decrypt messages received. The nature of the equipping they require depends on the <a href="/wiki/Encryption" title="Encryption">encryption</a> technique they might use. If they use a <a href="/wiki/Code_(cryptography)" title="Code (cryptography)">code</a>, both will require a copy of the same <a href="/wiki/Codebook" title="Codebook">codebook</a>. If they use a <a href="/wiki/Cipher" title="Cipher">cipher</a>, they will need appropriate keys. If the cipher is a <a href="/wiki/Symmetric_key" class="mw-redirect" title="Symmetric key">symmetric key cipher</a>, both will need a copy of the same key. If it is an <a href="/wiki/Public_key" class="mw-redirect" title="Public key">asymmetric key cipher</a> with the public/private key property, both will need the other's public key.
</p>

<h2><span class="mw-headline" id="Channel_of_exchange">Channel of exchange</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=1" title="Edit section: Channel of exchange">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Key exchange is done either in-band or out-of-band. encrypted.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="The_key_exchange_problem">The key exchange problem</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=2" title="Edit section: The key exchange problem">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The key exchange problem describes ways to exchange whatever keys or other information are needed for establishing a secure communication channel so that no one else can obtain a copy. Historically, before the invention of <a href="/wiki/Public-key_cryptography" title="Public-key cryptography">public-key cryptography</a> (asymmetrical cryptography), <a href="/wiki/Symmetric-key_algorithm" title="Symmetric-key algorithm">symmetric-key cryptography</a> utilized a single key to encrypt and decrypt messages. For two parties to communicate confidentially, they must first exchange the secret key so that each party is able to encrypt messages before sending, and decrypt received ones. This process is known as the key exchange.
</p><p>The overarching problem with symmetrical cryptography, or single-key cryptography, is that it requires a secret key to be communicated through trusted <a href="/wiki/Courier" title="Courier">couriers</a>, <a href="/wiki/Diplomatic_bag" title="Diplomatic bag">diplomatic bags</a>, or any other <a href="/wiki/Secure_channel" title="Secure channel">secure communication channel</a>. If two parties cannot establish a secure initial key exchange, they won't be able to communicate securely without the risk of messages being intercepted and decrypted by a third party who acquired the key during the initial key exchange.
</p><p>Public-key cryptography uses a two-key system, consisting of the public and the private keys, where messages are encrypted with one key and decrypted with another. It depends on the selected cryptographic algorithm which key—public or private—is used for encrypting messages, and which for decrypting. For example, in <a href="/wiki/RSA_(cryptosystem)" title="RSA (cryptosystem)">RSA</a>, the private key is used for decrypting messages, while in the <a href="/wiki/Digital_Signature_Algorithm" title="Digital Signature Algorithm">Digital Signature Algorithm</a> (DSA), the private key is used for encrypting them. The public key can be sent over non-secure channels or shared in public; the private key is only available to its owner.
</p><p>Known as the Diffie-Hellman key exchange, the encryption key can be openly communicated as it poses no risk to the confidentiality of encrypted messages. One party exchanges the keys to another party where they can then encrypt messages using the key and send back the cipher text. Only the decryption key—in this case, it's the private key—can decrypt that message. At no time during the Diffie-Hellman key exchange is any sensitive information at risk of compromise, as opposed to symmetrical key exchange.
</p>
<h3><span class="mw-headline" id="Identification">Identification</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=3" title="Edit section: Identification">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In principle, the only remaining problem was to be sure (or at least confident) that a public key actually belonged to its supposed owner. Because it is possible to '<a href="/wiki/Spoofing_attack" title="Spoofing attack">spoof</a>' another's identity in any of several ways, this is not a trivial or easily solved problem, particularly when the two users involved have never met and know nothing about each other.
</p>
<h3><span id="Diffie.E2.80.93Hellman_key_exchange"></span><span class="mw-headline" id="Diffie–Hellman_key_exchange">Diffie–Hellman key exchange</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=4" title="Edit section: Diffie–Hellman key exchange">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In 1976, Whitfield Diffie and Martin Hellman published a <a href="/wiki/Cryptographic_protocol" title="Cryptographic protocol">cryptographic protocol</a> called the <a href="/wiki/Diffie%E2%80%93Hellman_key_exchange" title="Diffie–Hellman key exchange">Diffie–Hellman key exchange</a> (D–H) based on concepts developed by Hellman's PhD student <a href="/wiki/Ralph_Merkle" title="Ralph Merkle">Ralph Merkle</a>. The protocol enables users to securely exchange secret keys even if an opponent is monitoring that communication channel. The D–H key exchange protocol, however, does not by itself address authentication (i.e. the problem of being sure of the actual identity of the person or 'entity' at the other end of the communication channel).  Authentication is crucial when an opponent can both monitor <i>and alter</i> messages within the communication channel (AKA <a href="/wiki/Man-in-the-middle_attack" title="Man-in-the-middle attack">man-in-the-middle</a> or MITM attacks) and was addressed in the fourth section of the paper.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Public_key_infrastructure">Public key infrastructure</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=5" title="Edit section: Public key infrastructure">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Public_key_infrastructure" title="Public key infrastructure">Public key infrastructures</a> (PKIs) have been proposed as a workaround for the problem of identity authentication. In their most usual implementation, each user applies to a “<a href="/wiki/Certificate_authority" title="Certificate authority">certificate authority</a>” (CA), trusted by all parties, for a <a href="/wiki/Digital_certificate" class="mw-redirect" title="Digital certificate">digital certificate</a> which serves for other users as a non-tamperable authentication of identity. The infrastructure is safe, unless the CA itself is compromised. In case it is, though, many PKIs provide a way to revoke certificates so other users will not trust them. Revoked certificates are usually put in <a href="/wiki/Certificate_revocation_list" title="Certificate revocation list">certificate revocation lists</a> which any certificate can be matched against.
</p><p>Several countries and other jurisdictions have passed <a href="/wiki/Legislation" title="Legislation">legislation</a> or issued regulations encouraging PKIs by giving (more or less) legal effect to these digital certificates (see <a href="/wiki/Digital_signature" title="Digital signature">digital signature</a>). Several commercial firms, and a few government departments, have established such certificate authorities. <a href="/wiki/VeriSign" class="mw-redirect" title="VeriSign">VeriSign</a> is the most prominent commercial firm providing digital identity services.
</p><p>This does nothing to solve the problem though, as the trustworthiness of the CA itself is still not guaranteed for any particular individual. It is a form of <a href="/wiki/Argument_from_authority" title="Argument from authority">argument from authority</a> fallacy. For actual trustworthiness, personal verification that the certificate belongs to the CA and establishment of trust in the CA are required. This is usually not possible.
</p><p>There are known cases where <a href="/wiki/Authoritarianism" title="Authoritarianism">authoritarian</a> governments proposed establishing so-called “national CAs” whose certificates would be mandatory to install on citizens’ devices and, once installed and trusted, could be used for monitoring, intercepting, modifying, or blocking the encrypted internet traffic.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup><sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup>
</p><p>For those new to such things, these arrangements are best thought of as electronic <a href="/wiki/Notary_public" title="Notary public">notary</a> endorsements that “this public key belongs to this user”. As with notary endorsements, there can be mistakes or misunderstandings in such vouchings. Additionally, the notary itself can be untrusted. There have been several high-profile public failures by assorted certificate authorities.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2017)">citation needed</span></a></i>&#93;</sup>
</p>
<h3><span class="mw-headline" id="Web_of_trust">Web of trust</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=6" title="Edit section: Web of trust">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>At the other end of the conceptual range is the <a href="/wiki/Web_of_trust" title="Web of trust">web of trust</a> system, which avoids central Certificate Authorities entirely. Each user is responsible for getting any certificate from another before using that certificate to communicate with, vet digital signatures from, ... the user claimed to be associated with the particular public key in a certificate. <a href="/wiki/Pretty_Good_Privacy" title="Pretty Good Privacy">PGP</a> (and <a href="/wiki/GNU_Privacy_Guard" title="GNU Privacy Guard">GPG</a>, an implementation of the <a href="/wiki/OpenPGP" class="mw-redirect" title="OpenPGP">OpenPGP</a> Internet Standard) employ just such a web of trust mechanism. Together they are the most widely used high quality cryptographic system in the world.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2012)">citation needed</span></a></i>&#93;</sup>
</p>
<h3><span class="mw-headline" id="Password-authenticated_key_agreement">Password-authenticated key agreement</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=7" title="Edit section: Password-authenticated key agreement">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Password-authenticated_key_agreement" title="Password-authenticated key agreement">Password-authenticated key agreement</a> algorithms can perform a cryptographic key exchange utilizing knowledge of a user's <a href="/wiki/Password" title="Password">password</a>.
</p>
<h3><span class="mw-headline" id="Quantum_key_exchange">Quantum key exchange</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Key_exchange&amp;action=edit&amp;section=8" title="Edit section: Quantum key exchange">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Quantum_key_distribution" title="Quantum key distribution">quantum key distribution</a></div>
<p><a href="/wiki/Quantum_key_distribution" title="Quantum key distribution">Quantum key distribution</a> exploits certain properties of quantum physics to ensure its security. It relies on the fact that observations (or <a href="/wiki/Measurement_in_quantum_mechanics" title="Measurement in quantum mechanics">measurements</a>) of a quantum state introduces perturbations in that state. Over many systems, these perturbations are detectable as noise by the receiver, making it possible to detect <a href="/wiki/Man-in-the-middle_attack" title="Man-in-the-middle attack">man-in-the-middle attacks</a>. Beside the <a href="/wiki/Correctness_(computer_science)" title="Correctness (computer science)">correctness</a> and <a href="/wiki/Completeness_(logic)" title="Completeness (logic)">completeness</a> of quantum mechanics, the protocol assumes the availability of an <a href="/wiki/Message_authentication" title="Message authentication">authenticated channel</a> between Alice and Bob.
</p>

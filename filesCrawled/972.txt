	<h1 id="firstHeading" class="firstHeading" lang="en">Kerberos (protocol)</h1>
	
	<div id="bodyContent" class="mw-body-content">
		<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>

<p><b>Kerberos</b> (<span class="nowrap"><span class="IPA nopopups noexcerpt"><a href="/wiki/Help:IPA/English" title="Help:IPA/English">/<span style="border-bottom:1px dotted"><span title="/ˈ/: primary stress follows">ˈ</span><span title="&#39;k&#39; in &#39;kind&#39;">k</span><span title="/ɜːr/: &#39;ur&#39; in &#39;fur&#39;">ɜːr</span><span title="&#39;b&#39; in &#39;buy&#39;">b</span><span title="/ər/: &#39;er&#39; in &#39;letter&#39;">ər</span><span title="/ɒ/: &#39;o&#39; in &#39;body&#39;">ɒ</span><span title="&#39;s&#39; in &#39;sigh&#39;">s</span></span>/</a></span></span>) is a <a href="/wiki/Computer_network" title="Computer network">computer-network</a> <a href="/wiki/Authentication" title="Authentication">authentication</a> <a href="/wiki/Cryptographic_protocol" title="Cryptographic protocol">protocol</a> that works on the basis of <i>tickets</i> to allow <a href="/wiki/Node_(networking)" title="Node (networking)">nodes</a> communicating over a non-secure network to prove their identity to one another in a secure manner. The protocol was named after the character <i><a href="/wiki/Cerberus" title="Cerberus">Kerberos</a></i> (or <i><a href="/wiki/Cerberus" title="Cerberus">Cerberus</a></i>) from <a href="/wiki/Greek_mythology" title="Greek mythology">Greek mythology</a>, the ferocious three-headed guard dog of <a href="/wiki/Hades" title="Hades">Hades</a>. Its designers aimed it primarily at a <a href="/wiki/Client%E2%80%93server" class="mw-redirect" title="Client–server">client–server</a> model and it provides <a href="/wiki/Mutual_authentication" title="Mutual authentication">mutual authentication</a>—both the user and the server verify each other's identity. Kerberos protocol messages are protected against <a href="/wiki/Computer_insecurity#Eavesdropping" class="mw-redirect" title="Computer insecurity">eavesdropping</a> and <a href="/wiki/Replay_attack" title="Replay attack">replay attacks</a>.
</p><p>Kerberos builds on <a href="/wiki/Symmetric_key_cryptography" class="mw-redirect" title="Symmetric key cryptography">symmetric key cryptography</a> and requires a <a href="/wiki/Trusted_third_party" title="Trusted third party">trusted third party</a>, and optionally may use <a href="/wiki/Public-key_cryptography" title="Public-key cryptography">public-key cryptography</a> during certain phases of authentication.<sup id="cite_ref-rfc4556_1-0" class="reference"><a href="#cite_note-rfc4556-1">&#91;1&#93;</a></sup> Kerberos uses UDP port 88 by default.
</p>

<h2><span class="mw-headline" id="History_and_development">History and development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=1" title="Edit section: History and development">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Massachusetts_Institute_of_Technology" title="Massachusetts Institute of Technology">Massachusetts Institute of Technology</a> (MIT) developed Kerberos to protect network services provided by <a href="/wiki/Project_Athena" title="Project Athena">Project Athena</a>.<sup id="cite_ref-Athena_Network_Services_2-0" class="reference"><a href="#cite_note-Athena_Network_Services-2">&#91;2&#93;</a></sup><sup id="cite_ref-Building_Internet_Firewalls_3-0" class="reference"><a href="#cite_note-Building_Internet_Firewalls-3">&#91;3&#93;</a></sup>  The protocol is based on the earlier <a href="/wiki/Needham%E2%80%93Schroeder_protocol" title="Needham–Schroeder protocol">Needham–Schroeder symmetric key protocol</a>. Several versions of the protocol exist; versions 1–3 occurred only internally at MIT.
</p><p>Kerberos version 4 was primarily designed by <a href="/w/index.php?title=Steven_P_Miller_(software_designer)&amp;action=edit&amp;redlink=1" class="new" title="Steven P Miller (software designer) (page does not exist)">Steve Miller</a> and <a href="/w/index.php?title=Clifford_Neuman&amp;action=edit&amp;redlink=1" class="new" title="Clifford Neuman (page does not exist)">Clifford Neuman</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup>  Published in the late 1980s, version 4 was also targeted at <a href="/wiki/Project_Athena" title="Project Athena">Project Athena</a>.
</p><p>Neuman and Kohl published version 5 in 1993 with the intention of overcoming existing limitations and security problems.  Version 5 appeared as <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc1510">RFC 1510</a>, and was made obsolete by <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4120">RFC 4120</a> in 2005. 
</p><p>Authorities in the <a href="/wiki/United_States" title="United States">United States</a> classified Kerberos as "Auxiliary Military Equipment" on the US Munitions List and banned its <a href="/wiki/Export_of_cryptography_in_the_United_States" class="mw-redirect" title="Export of cryptography in the United States">export</a> because it used the <a href="/wiki/Data_Encryption_Standard" title="Data Encryption Standard">Data Encryption Standard</a> (DES) <a href="/wiki/Encryption_algorithm" class="mw-redirect" title="Encryption algorithm">encryption algorithm</a> (with 56-bit keys). A non-US Kerberos 4 implementation, KTH-KRB developed at the <a href="/wiki/Royal_Institute_of_Technology" class="mw-redirect" title="Royal Institute of Technology">Royal Institute of Technology</a> in <a href="/wiki/Sweden" title="Sweden">Sweden</a>, made the system available outside the US before the US changed its <a href="/wiki/Export_of_cryptography_in_the_United_States" class="mw-redirect" title="Export of cryptography in the United States">cryptography export</a> regulations (<i>circa</i> 2000). The Swedish implementation was based on a limited version called eBones. eBones was based on the exported MIT Bones release (stripped of both the encryption functions and the calls to them) based on version Kerberos 4 patch-level 9.
</p><p>In 2005, the <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">Internet Engineering Task Force</a> (IETF) Kerberos working group updated specifications. Updates included:
</p>
<ul><li>Encryption and Checksum Specifications (<a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc3961">RFC 3961</a>).</li>
<li><a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">Advanced Encryption Standard</a> (AES) Encryption for Kerberos 5 (<a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc3962">RFC 3962</a>).</li>
<li>A new edition of the Kerberos V5 specification "The Kerberos Network Authentication Service (V5)" (<a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4120">RFC 4120</a>). This version obsoletes <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc1510">RFC 1510</a>, clarifies aspects of the protocol and intended use in a more detailed and clearer explanation.</li>
<li>A new edition of the <a href="/wiki/Generic_Security_Services_Application_Program_Interface" title="Generic Security Services Application Program Interface">Generic Security Services Application Program Interface</a> (GSS-API) specification "The Kerberos Version 5 Generic Security Service Application Program Interface (GSS-API) Mechanism: Version 2." (<a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4121">RFC 4121</a>).</li></ul>
<p>MIT makes an implementation of Kerberos freely available, under copyright permissions similar to those used for <a href="/wiki/BSD_licenses" title="BSD licenses">BSD</a>. In 2007, MIT formed the Kerberos Consortium to foster continued development. Founding sponsors include vendors such as <a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle</a>, <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple Inc.</a>, <a href="/wiki/Google" title="Google">Google</a>, <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a>, Centrify Corporation and <a href="/w/index.php?title=TeamF1&amp;action=edit&amp;redlink=1" class="new" title="TeamF1 (page does not exist)">TeamF1 Inc.</a>, and academic institutions such as the <a href="/wiki/Royal_Institute_of_Technology" class="mw-redirect" title="Royal Institute of Technology">Royal Institute of Technology</a> in Sweden, Stanford University, MIT, and vendors such as CyberSafe offering commercially supported versions.
</p>
<h2><span class="mw-headline" id="Microsoft_Windows">Microsoft Windows</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=2" title="Edit section: Microsoft Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> and later versions use Kerberos as its default authentication method.<sup id="cite_ref-MSTN-Kerberos-Auth_5-0" class="reference"><a href="#cite_note-MSTN-Kerberos-Auth-5">&#91;5&#93;</a></sup> Some <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> additions to the Kerberos suite of protocols are documented in <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc3244">RFC 3244</a> "Microsoft Windows 2000 Kerberos Change Password and Set Password Protocols". <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4757">RFC 4757</a> documents Microsoft's use of the <a href="/wiki/RC4" title="RC4">RC4</a> cipher.  While Microsoft <a href="/wiki/Embrace,_extend,_and_extinguish" title="Embrace, extend, and extinguish">uses and extends</a> the Kerberos protocol, it does not use the MIT software.
</p><p>Kerberos is used as preferred authentication method:
In general, joining a client to a Windows domain means enabling Kerberos as default protocol for authentications from that client to services in the Windows domain and all domains with trust relationships to that domain.<sup id="cite_ref-MSTN-Kerberos-Auth_5-1" class="reference"><a href="#cite_note-MSTN-Kerberos-Auth-5">&#91;5&#93;</a></sup>
</p><p>In contrast, when either client or server or both are not joined to a domain (or not part of the same trusted domain environment), Windows will instead use <a href="/wiki/NTLM" class="mw-redirect" title="NTLM">NTLM</a> for authentication between client and server.<sup id="cite_ref-MSTN-Kerberos-Auth_5-2" class="reference"><a href="#cite_note-MSTN-Kerberos-Auth-5">&#91;5&#93;</a></sup>
</p><p>Intranet web applications can enforce Kerberos as an authentication method for domain joined clients by using APIs provided under <a href="/wiki/SSPI" class="mw-redirect" title="SSPI">SSPI</a>.
</p>
<h2><span class="mw-headline" id="UNIX_and_Unix-like_operating_systems">UNIX and Unix-like operating systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=3" title="Edit section: UNIX and Unix-like operating systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Many UNIX and UNIX-like operating systems, including <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a>, Apple's <a href="/wiki/MacOS" title="MacOS">macOS</a>, <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>, <a href="/wiki/Sun_microsystems" class="mw-redirect" title="Sun microsystems">Oracle</a>'s <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a>, IBM's <a href="/wiki/AIX" class="mw-redirect" title="AIX">AIX</a> and <a href="/wiki/Z/OS" title="Z/OS">Z/OS</a>, HP's <a href="/wiki/HP-UX" title="HP-UX">HP-UX</a> and <a href="/wiki/OpenVMS" title="OpenVMS">OpenVMS</a> and others, include software for Kerberos authentication of users or services. Embedded implementation of the Kerberos V authentication protocol for client agents and network services running on embedded platforms is also available from companies.
</p>
<h2><span class="mw-headline" id="Protocol">Protocol</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=4" title="Edit section: Protocol">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=5" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The client authenticates itself to the <b>Server (AS)</b> which forwards the username to a <b><a href="/wiki/Key_distribution_center" title="Key distribution center">key distribution center</a></b> <b>(KDC)</b>. The KDC issues a <b>ticket-granting ticket (TGT)</b>, which is time stamped and encrypts it using the <b>ticket-granting service's (TGS)</b> secret key and returns the encrypted result to the user's workstation.  This is done infrequently, typically at user logon; the TGT expires at some point although it may be transparently renewed by the user's session manager while they are logged in.
</p><p>When the client needs to communicate with a service on another node (a "principal", in Kerberos parlance), the client sends the TGT to the TGS, which usually shares the same host as the KDC. Service must be registered at TGT with a <b>Service Principal Name (SPN)</b>. The client uses the SPN to request access to this service. After verifying that the TGT is valid and that the user is permitted to access the requested service, the TGS issues ticket and session keys to the client. The client then sends the ticket to the <b>service server (SS)</b> along with its service request.
</p>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Kerberos.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/4/4e/Kerberos.svg/220px-Kerberos.svg.png" decoding="async" width="220" height="173" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/4/4e/Kerberos.svg/330px-Kerberos.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/4/4e/Kerberos.svg/440px-Kerberos.svg.png 2x" data-file-width="444" data-file-height="350" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Kerberos.svg" class="internal" title="Enlarge"></a></div>Kerberos negotiations</div></div></div>
<p>The protocol is described in detail below.
</p>
<h4><span class="mw-headline" id="User_Client-based_Login">User Client-based Login</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=6" title="Edit section: User Client-based Login">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ol><li>A user enters a username and password on the <a href="/wiki/Client_(computing)" title="Client (computing)">client machine(s)</a>.  Other credential mechanisms like pkinit (<a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4556">RFC 4556</a>) allow for the use of public keys in place of a password.</li>
<li>The client transforms the password into the key of a symmetric cipher.  This either uses the built-in key scheduling, or a <a href="/wiki/One-way_hash" class="mw-redirect" title="One-way hash">one-way hash</a>, depending on the cipher-suite used.</li></ol>
<h4><span class="mw-headline" id="Client_Authentication">Client Authentication</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=7" title="Edit section: Client Authentication">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ol><li>The client sends a <a href="/wiki/Cleartext" class="mw-redirect" title="Cleartext">cleartext</a> message of the user ID to the AS (Authentication Server) requesting services on behalf of the user. (Note: Neither the secret key nor the password is sent to the AS.)</li>
<li>The AS checks to see if the client is in its database. If it is, the AS generates the secret key by hashing the password of the user found at the database (e.g., <a href="/wiki/Active_Directory" title="Active Directory">Active Directory</a> in Windows Server) and sends back the following two messages to the client:
<ul><li>Message A: <i>Client/TGS Session Key</i> encrypted using the secret key of the client/user.</li>
<li>Message B: <i>Ticket-Granting-Ticket</i> (TGT, which includes the client ID, client <a href="/wiki/Network_address" title="Network address">network address</a>, ticket validity period, and the <i>client/TGS session key</i>) encrypted using the secret key of the TGS.</li></ul></li>
<li>Once the client receives messages A and B, it attempts to decrypt message A with the secret key generated from the password entered by the user.  If the user entered password does not match the password in the AS database, the client's secret key will be different and thus unable to decrypt message A.  With a valid password and secret key the client decrypts message A to obtain the <i>Client/TGS Session Key</i>. This session key is used for further communications with the TGS. (Note: The client cannot decrypt Message B, as it is encrypted using TGS's secret key.)  At this point, the client has enough information to authenticate itself to the TGS.</li></ol>
<h4><span class="mw-headline" id="Client_Service_Authorization">Client Service Authorization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=8" title="Edit section: Client Service Authorization">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ol><li>When requesting services, the client sends the following messages to the TGS:
<ul><li>Message C: Composed of the TGT from message B and the ID of the requested service.</li>
<li>Message D: Authenticator (which is composed of the client ID and the timestamp), encrypted using the <i>Client/TGS Session Key</i>.</li></ul></li>
<li>Upon receiving messages C and D, the TGS retrieves message B out of message C. It decrypts message B using the TGS secret key. This gives it the "client/TGS session key". Using this key, the TGS decrypts message D (Authenticator) and compare client ID from message C and D, if they match server sends the following two messages to the client:
<ul><li>Message E: <i>Client-to-server ticket</i> (which includes the client ID, client network address, validity period and <i>Client/Server Session Key</i>) encrypted using the service's secret key.</li>
<li>Message F: <i>Client/Server Session Key</i> encrypted with the <i>Client/TGS Session Key</i>.</li></ul></li></ol>
<h4><span class="mw-headline" id="Client_Service_Request">Client Service Request</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=9" title="Edit section: Client Service Request">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ol><li>Upon receiving messages E and F from TGS, the client has enough information to authenticate itself to the Service Server (SS).  The client connects to the SS and sends the following two messages:
<ul><li>Message E: from the previous step (the <i>client-to-server ticket</i>, encrypted using service's secret key).</li>
<li>Message G: a new Authenticator, which includes the client ID, timestamp and is encrypted using <i>Client/Server Session Key</i>.</li></ul></li>
<li>The SS decrypts the ticket (message E) using its own secret key to retrieve the <i>Client/Server Session Key</i>. Using the sessions key, SS decrypts the Authenticator and compares client ID from messages E and G, if they match server sends the following message to the client to confirm its true identity and willingness to serve the client:
<ul><li>Message H: the timestamp found in client's Authenticator (plus 1 in version 4, but not necessary in version 5<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup><sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup>), encrypted using the <i>Client/Server Session Key</i>.</li></ul></li>
<li>The client decrypts the confirmation (message H) using the <i>Client/Server Session Key</i> and checks whether the timestamp is correct. If so, then the client can trust the server and can start issuing service requests to the server.</li>
<li>The server provides the requested services to the client.</li></ol>
<h2><span class="mw-headline" id="Drawbacks_and_limitations">Drawbacks and limitations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=10" title="Edit section: Drawbacks and limitations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul><li>Single point of failure: It requires continuous availability of a central server. When the Kerberos server is down, new users cannot log in. This can be mitigated by using multiple Kerberos servers and fallback authentication mechanisms.</li>
<li>Kerberos has strict time requirements, which means the clocks of the involved hosts must be synchronized within configured limits. The tickets have a time availability period and if the host clock is not synchronized with the Kerberos server clock, the authentication will fail. The default configuration <a rel="nofollow" class="external text" href="http://web.mit.edu/Kerberos/krb5-1.5/krb5-1.5.4/doc/krb5-admin/Clock-Skew.html">per MIT</a> requires that clock times be no more than five minutes apart.  In practice <a href="/wiki/Network_Time_Protocol" title="Network Time Protocol">Network Time Protocol</a> daemons are usually used to keep the host clocks synchronized. Note that some servers (Microsoft's implementation being one of them) may return a KRB_AP_ERR_SKEW result containing the encrypted server time in case both clocks have an offset greater than the configured maximum value. In that case, the client could retry by calculating the time using the provided server time to find the offset. This behavior is documented in <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc4430">RFC 4430</a>.</li>
<li>The administration protocol is not standardized and differs between server implementations.  Password changes are described in <a class="external mw-magiclink-rfc" rel="nofollow" href="https://tools.ietf.org/html/rfc3244">RFC 3244</a>.</li>
<li>In case of symmetric cryptography adoption (Kerberos can work using symmetric or asymmetric (public-key) cryptography), since all authentications are controlled by a centralized <a href="/wiki/Key_distribution_center" title="Key distribution center">key distribution center</a> (KDC), compromise of this authentication infrastructure will allow an attacker to impersonate any user.</li>
<li>Each network service which requires a different host name will need its own set of Kerberos keys. This complicates virtual hosting and clusters.</li>
<li>Kerberos requires user accounts, user clients and the services on the server to all have a trusted relationship to the Kerberos token server (All must be in the same Kerberos domain or in domains that have a trust relationship between each other). Kerberos cannot be used in scenarios where users want to connect to services from unknown/untrusted clients as in a typical Internet or cloud computer scenario, where the authentication provider typically does not have knowledge about the users client system.</li>
<li>The required client trust makes creating staged environments (e.g., separate domains for test environment, pre-production environment and production environment) difficult: Either domain trust relationships need to be created that prevent a strict separation of environment domains or additional user clients need to be provided for each environment.</li></ul>
<h2><span class="mw-headline" id="Vulnerabilities">Vulnerabilities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Kerberos_(protocol)&amp;action=edit&amp;section=11" title="Edit section: Vulnerabilities">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Data_Encryption_Standard" title="Data Encryption Standard">Data Encryption Standard</a> (DES) cipher can be used in combination with Kerberos, but is no longer an Internet standard because it is weak.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup>  Security vulnerabilities exist in many legacy products that implement Kerberos because they have not been updated to use newer ciphers like AES instead of DES.
</p><p>In November 2014, Microsoft released a patch (MS14-068) to rectify an exploitable vulnerability in Windows implementation of the Kerberos Key Distribution Center (KDC).<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup> The vulnerability purportedly allows users to "elevate" (and abuse) their privileges, up to Domain level.
</p>
